Mascope development priorities.

### Backend:
1. linux/windows support:
	Linux support is essential for better performance, scalability (containerization) and all kinds of automation, since testing/CI/CD automation runs in virtual environments, where windows is not a good choice.
	a) parse sources and scripts to linux (Ubuntu);
	b) VBox is the primary deployment media: create scripts for VBox deployment automation;
	c) docker containerization is a further step in deployment development. It provides better scalability, resource management, faster deployment. But this bullet is more of a future challenge due to missing or unclear specs: e.g. big data sharing (network file system, cloud store, shared volumes...), service bundles as deployment/distribution model still to be developed, etc.
2. Backend test automation:
	Lack of test automation gives the way to re-accuring bugs into the code - many of them are really obscure and take quite long to find and fix: we are heavily in need of regression testing, system tests and integration tests:
	a) unittest infrastructure is implemented, now we need to use it more actively;
	b) system/integration testing: automation media must be defined here. Robotic automation (robotframework via Mascope UI) is a standard choice, but it can be overly heavy for current stage of the project. We can try implementing standard reproducible scenarios by pure backend gearing, then extract+measure+compare results:
		- implement test library + test code;
		- ensure reproducibility of test scenarios;
		- create corresponding test environments and contexts (scripting);
		- run tests, collect and store test metrics;
		- compare test results with previous runs;
		- integrate this process into git CI/CD pipeline;
3. Production deployment improvements:
	a) use nginx as web_server/balance_loader with karsa-router-service. This is a must for production deployment, but for the moment this can be postponed: we are not clear yet about backend production deployment details; then if karsa-router-service stars to be a bottleneck, we may need another server socket (another instance of karsa-router) - this should be understood first.
	b) karsa-router-service uses http polling: switching to web sockets may ensure quicker and more stable communication between services; yet this is not a top priority - rather improvement item, since current model works ok;
4. New product features:
	a) service status: this is a feature request, which should be designed well -- we need more use cases to come up with requirements and finally specs:
		- show service statuses in UI;
		- update service workflows for consistent behaviour with missing service dependencies;
		- ...
5. Implement continuous profiling.
	Continuous profiling provides means for continuous performance tracing during development process -- so, as such, not practical without system/integration test automation (bullets 1, 2). 
	a) We have t_mark telemark profiling: we need to implement telemarking into all standard workflows to get traceable telemetry for every code change;
	b) Memory management/footprint/leak detection - TODO
	c) ...
6. CI/CD pipeline
   As soon as we are clear with bullets 1 and 2, this item can be implemented via git resources.
